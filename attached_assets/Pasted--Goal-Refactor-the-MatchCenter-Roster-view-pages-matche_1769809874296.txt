## Goal

Refactor the MatchCenter ‚ÄúRoster‚Äù view (@pages/matches/[id]/matchcenter/index.tsx) so that:

- `RosterTab` is dissolved and its responsibilities are moved into `MatchCenter`.
- `RosterList` becomes the rich, single‚Äëteam roster table used directly inside `MatchCenter`.

## 1. RosterTab removal and integration into MatchCenter

1. Identify current usages of `RosterTab` in `MatchCenter`, especially the `activeTab === "roster"` block.
2. Remove `RosterTab` and inline its logic into `MatchCenter`:
   - Keep the sorting behavior: positions ordered C, A, G, F, then by jersey number.
   - Implement a shared `sortRoster` utility (e.g. `utils/matchRoster.tsx`) that takes `RosterPlayer[]` and returns the sorted list; use the same function for home and away rosters to avoid duplication.
   - For the roster tab content, render two `RosterList` instances side by side (home/away) as currently done in `RosterTab`, but directly from `MatchCenter`.
3. Ensure existing player stats logic (`homePlayerStats`, `awayPlayerStats` computed in a `useEffect`) is still passed into each `RosterList`.
4. Verify that routing and permissions behavior remains unchanged:
   - `permissions.showButtonRosterHome` and `permissions.showButtonRosterAway` still control whether the ‚ÄúEdit roster‚Äù action is available.
   - Roster tab remains the default `activeTab === "roster"`.

## 3. New roster validation feature

### 3.1 Endpoint

- For home:
  - `POST` `/matches/{_id}/home/roster/validate`
- For away:
  - Presumably `/matches/{_id}/away/roster/validate` with the same contract (mirrored for away team).
- Endpoint is authenticated with the current user (reuse existing `apiClient` configuration).

### 3.2 Frontend flow (MatchCenter)

- Add per‚Äëteam `isValidatingHomeRoster` and `isValidatingAwayRoster` booleans in `MatchCenter` state.
- Pass these down to `RosterList` as `isValidating`.
- Provide per‚Äëteam `onValidateRoster` callbacks to `RosterList`:
  - Home:
    - Set `isValidatingHomeRoster` to true.
    - Call backend endpoint `/matches/{match._id}/home/roster/validate`.
    - The response returns a `players` array with updated `eligibilityStatus` values.
    - Merge updated roster data into `match.home.roster.players` and update `match` state via `setMatch`.
    - Set `isValidatingHomeRoster` back to false.
  - Away:
    - Same pattern for `/away/roster/validate`.
- While `isValidatingX` is true:
  - Show spinner in the eligibility status column cells.
  - Optionally disable the ‚ÄúPr√ºfen‚Äù button to avoid duplicate calls.

### 3.3 Permissions

- `canValidateRoster` in `RosterList` should be derived from `permissions` in `MatchCenter`. If user can edit roster he also can validate it.
- If the user is not allowed to validate:
  - Hide the ‚ÄúPr√ºfen‚Äù button entirely (no disabled state needed).

## 4. Player‚ÄëCard Modal: Data and triggers

Primary purpose: identity verification via photo, plus quick view on licenses and season stats.

### 4.1 Triggers

From each roster table row, the modal opens when:

- The Magnifying‚ÄëGlass icon is clicked.
- The player name is clicked.
- The player avatar is clicked.

Implementation:

- `RosterList` receives `onOpenPlayerCard`.
- Inside `RosterList`, wrap avatar, name, and icon with onClick handlers that call `onOpenPlayerCard({ playerId, teamFlag, player })`.
- In `MatchCenter`, maintain a `selectedPlayer` state and `isPlayerCardOpen` boolean.
  - On `onOpenPlayerCard`, set `selectedPlayerContext` and open the modal.
  - Use `Dialog` and `Transition` from `@headlessui/react` (already used in the file) to implement the modal.

### 4.2 Data loading

- Minimal data for the header (name, jersey number, position, team, basic eligibility status) can be passed directly from the roster row (no additional fetch required).
- For extended data:
  - Use existing `/players/{playerId}` endpoint already invoked in MatchCenter for `calledMatches`.
  - Either reuse already fetched player data from `fetchPlayerStats` (if available) or fetch detailed player data on demand when opening the modal.
- The modal component should accept:
  - `playerContext` (includes roster row data, team name, team logo, team label like ‚Äú1. Herren‚Äù).
  - `playerDetails` (optional, asynchronous details from the player API).
  - `onClose`.
